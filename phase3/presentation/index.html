Vulnerabilities

... and how we tried to fix them

Core of presentation: at this point in the presentation you must talk about: Vulnerabilities (possible exploits of the vulnerabilities) and how you fixed them. Feel free to go into technical details, but keep it short, clear and entertaining. Report possible changes in OWASP and CVSS values (e.g. from high to medium, etc.) including reasoning for these changes in the values.

The HTTPS Migration

Moving the website from HTTP to HTTPS solved a large amount of issues that related to clear text data.

Moving the DocumentRoot of GitHUB

One of the major vulnerabilities was the fact that a few files were not proteceted, these files included sensitive infomration regarding the setup of our machine. Our solution was to change the github DocumentRoot to the folder that only contains the PHP code.

Timing Attack

One of the vulnerbilities was a timing attack, where the login page would reply within 50ms if the user was registered and within around 25mn if the user was not registered. While minor, this attack could be used to check whether a certain email is registered with gnb or not. To overcome this a random waiting period was introduced to all login attemps, making the response time less uniform and more random.

Unsanitized Input

Creating a unified sanitizing function was the solution to combating unsanitizaiton.

Integer overflow

This was due to a comparison between a long long value and an integer value inside the c parser. By changing the integer to a long long value, this is no longer possible

Clickjacking

This has been rectified through the updated security policy and The X-FRAME-OPTIONS has been set to DENY.
